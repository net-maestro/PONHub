1. ВВЕДЕНИЕ И ПОСТАНОВКА ПРОБЛЕМЫ
--------------------------------------------------------------------------------

В современных сетях широкополосного доступа технологии пассивных оптических
сетей (GPON — Gigabit-capable Passive Optical Network) занимают доминирующее
положение. Согласно данным аналитической компании Dell'Oro Group, к 2025 году
более 75% новых подключений фиксированного широкополосного доступа реализуются
именно по технологии GPON. Ключевым элементом такой сети является оптическая
линейный терминал (OLT — Optical Line Terminal), управляющий множеством
абонентских устройств (ONU — Optical Network Unit).

Процесс регистрации и настройки нового абонентского оборудования традиционно
требует от инженера оператора выполнения последовательности CLI-команд
(Command Line Interface) через терминальное соединение с оборудованием OLT.
Данная процедура характеризуется следующими проблемами:

1) Высокая вероятность человеческой ошибки. Ручной ввод серийных номеров,
   IP-адресов, идентификаторов VLAN и ONU-ID сопряжён с риском
   синтаксических и логических ошибок, что приводит к неработоспособности
   подключения или конфликтам в сети.

2) Низкая скорость обслуживания. Типовая процедура регистрации одной ONU
   занимает от 10 до 20 минут при ручном выполнении, включая получение
   свободных адресов, проверку конфигурации порта и ввод команд.

3) Отсутствие стандартизации. Разные операторы и инженеры используют
   различные подходы к конфигурированию, что усложняет аудит, поддержку
   и масштабирование сети.

4) Отсутствие визуального контроля. Традиционный CLI-подход не предоставляет
   оператору возможности предварительного просмотра конфигурации до её
   применения, что исключает возможность верификации на этапе планирования.

Проект PONHub представляет собой комплексное решение данной проблемы,
включающее веб-ориентированную клиентскую часть (Frontend), реализованную
на базе современного стека технологий Vue.js 3, обеспечивающую интуитивный
интерфейс для автоматизации процесса регистрации ONU на оборудовании ZTE.

2. АНАЛИЗ СУЩЕСТВУЮЩИХ РЕШЕНИЙ И ОБОСНОВАНИЕ ВЫБОРА ТЕХНОЛОГИЙ
--------------------------------------------------------------------------------

2.1. Обзор существующих подходов к управлению ONU

На текущий момент существуют следующие подходы к управлению абонентским
оборудованием в сетях GPON:

а) Прямое CLI-управление через Telnet/SSH. Классический подход, при котором
   инженер подключается к OLT посредством терминального эмулятора (PuTTY,
   SecureCRT и др.) и вручную вводит команды. Недостатки: высокая
   трудоёмкость, отсутствие контроля версий конфигураций, невозможность
   предварительной валидации.

б) Проприетарные системы управления вендоров. Производители оборудования
   (ZTE, Huawei, Nokia) предлагают собственные системы сетевого управления
   (NMS — Network Management System), такие как ZTE NetNumen. Однако эти
   решения часто отличаются высокой стоимостью, избыточной функциональностью
   для задач малого и среднего бизнеса, а также закрытой архитектурой,
   затрудняющей интеграцию с внешними системами.

в) Скриптовые решения на базе Python/Bash. Многие операторы разрабатывают
   собственные скрипты для автоматизации. Такие решения, как правило, не
   имеют пользовательского интерфейса, требуют технических знаний для
   использования и трудно масштабируются.

г) Веб-ориентированные системы управления. Современный подход, предполагающий
   создание одностраничных веб-приложений (SPA — Single Page Application),
   обеспечивающих удобный доступ к функционалу автоматизации через браузер.

Проект PONHub относится к последней категории и сочетает преимущества
веб-доступности с глубокой интеграцией с сетевым оборудованием.

2.2. Обоснование выбора технологического стека Frontend

Для реализации клиентской части PONHub были рассмотрены следующие
фреймворки: React, Angular и Vue.js 3.

React (Meta Platforms) — наиболее популярная библиотека для построения UI.
Преимущества: огромная экосистема, высокая производительность виртуального
DOM. Недостатки: необходимость подключения дополнительных библиотек для
управления состоянием (Redux, MobX), более сложный порог входа для
начинающих разработчиков.

Angular (Google) — полноценный фреймворк с встроенными решениями для
маршрутизации, форм, HTTP-клиента. Преимущества: строгая типизация
(TypeScript), мощная экосистема. Недостатки: высокая сложность,
избыточность для проектов среднего размера, крутая кривая обучения.

Vue.js 3 — прогрессивный фреймворк, сочетающий простоту использования с
мощными возможностями Composition API. Преимущества: низкий порог входа,
отличная производительность, встроенная реактивность через Proxy-объекты,
лёгкая интеграция с существующими проектами.

Выбор Vue.js 3 для проекта PONHub обусловлен следующими факторами:

1) Composition API. Новый API Vue.js 3 позволяет организовать логику
   компонента вокруг бизнес-задач, а не опций (data, methods, computed),
   что улучшает читаемость и переиспользование кода. Для системы управления
   сетевым оборудованием, где компоненты имеют сложную бизнес-логику
   (валидация данных, предварительный просмотр конфигурации, обработка
   статусов), это критически важно.

2) Реактивная система на базе Proxy. Vue.js 3 использует JavaScript Proxy
   для реализации реактивности, что обеспечивает более эффективное
   отслеживание изменений по сравнению с Object.defineProperty во Vue.js 2.
   Для интерфейса, работающего с данными о множестве ONU, портов OLT и
   параметров сети, это гарантирует плавную работу интерфейса.

3) Pinia — официальное решение для управления состоянием. Pinia, заменившая
   Vuex во Vue.js экосистеме, предоставляет простой и типобезопасный API
   для глобального управления состоянием. В PONHub Pinia используется для
   хранения данных о сессиях, конфигурациях OLT и статусов операций.

4) Vite — современный сборщик. Vite обеспечивает мгновенный запуск
   дев-сервера благодаря использованию нативных ES-модулей браузера,
   а также оптимизированную продакшен-сборку через Rollup. Это ускоряет
   процесс разработки и отладки frontend-части.

5) Bootstrap 5. Выбор Bootstrap в качестве UI-фреймворка обусловлен его
   готовой компонентной базой (таблицы, формы, модальные окна, уведомления),
   адаптивностью и совместимостью с Vue.js через библиотеку BootstrapVue
   или прямое использование CSS-классов.

3. АРХИТЕКТУРА FRONTEND-ЧАСТИ СИСТЕМЫ PONHub
--------------------------------------------------------------------------------

3.1. Общая архитектура Single Page Application

Frontend PONHub реализован как одностраничное приложение (SPA), что
подразумевает загрузку единой HTML-страницы и динамическое обновление
контента через JavaScript без перезагрузки страницы. Архитектура включает
следующие уровни:

Уровень 1: Представление (Views/Components)
   — Страницы (views): главные экраны приложения (главная панель, страница
     регистрации ONU, страница мониторинга, настройки)
   — Компоненты (components): переиспользуемые UI-элементы (формы ввода,
     таблицы данных, карточки ONU, индикаторы состояния)

Уровень 2: Управление состоянием (State Management — Pinia)
   — Хранилища (stores): глобальное состояние приложения (список OLT,
     данные ONU, статусы операций, пользовательские сессии)

Уровень 3: Маршрутизация (Vue Router)
   — Определение маршрутов (routes) и защита переходов (guards) для
     управления навигацией и контролем доступа

Уровень 4: Сетевое взаимодействие (Axios/_fetch)
   — HTTP-клиент для взаимодействия с REST API Backend

Уровень 5: Утилиты и вспомогательные модули
   — Функции валидации, форматирования данных, обработки ошибок

3.2. Компонентная архитектура

Принцип компонентной архитектуры Vue.js предполагает разделение интерфейса
на независимые, переиспользуемые модули. В PONHub реализованы следующие
ключевые компоненты:

а) Компонент формы ввода данных ONU (OnuRegistrationForm)
   Предназначение: сбор и валидация входных данных — серийного номера (SN),
   типа ONU, VLAN ID.
   
   Особенности:
   — Реактивная валидация в реальном времени с использованием встроенных
     валидаторов Vue.js
   — Динамическое отображение полей в зависимости от выбранного типа ONU
   — Визуальная индикация ошибок ввода (подсветка полей, подсказки)
   — Автоматическая проверка формата SN (регулярные выражения)

б) Компонент предварительного просмотра конфигурации (ConfigPreview)
   Предназначение: отображение сгенерированных CLI-команд до их применения.
   
   Особенности:
   — Подсветка синтаксиса CLI-команд для повышения читаемости
   — Группировка команд по этапам выполнения (настройка порта, регистрация
     ONU, назначение IP, настройка VLAN)
   — Визуальное предупреждение о потенциальных конфликтах
   — Возможность копирования команд в буфер обмена

в) Компонент мониторинга ONU (OnuMonitor)
   Предназначение: отображение информации о зарегистрированных ONU, включая
   MAC-адреса подключённых клиентских устройств.
   
   Особенности:
   — Табличное представление с возможностью сортировки и фильтрации
   — Автообновление данных через периодические запросы к API
   — Цветовая индикация статуса ONU (онлайн/офлайн)
   — Детальная карточка ONU с полной информацией о конфигурации

г) Компонент управления OLT (OltManager)
   Предназначение: отображение и управление списком OLT-узлов.
   
   Особенности:
   — Карточки OLT с информацией о статусе, количестве зарегистрированных
     ONU, загрузке портов
   — Возможность добавления/редактирования конфигурации OLT
   — Визуализация структуры портов (PON-порты, ONU на каждом порту)

3.3. Управление состоянием через Pinia

Pinia — официальная библиотека управления состоянием для Vue.js 3,
заменяющая Vuex. В PONHub хранилища Pinia организованы следующим образом:

a) OltStore — состояние, связанное с OLT-оборудованием:
   — Список доступных OLT (из конфигурационного файла YAML)
   — Статус подключения к каждому OLT
   — Данные о портах и зарегистрированных ONU

b) OnuStore — состояние регистрации ONU:
   — Текущая форма ввода (SN, тип, VLAN)
   — Результат предварительного просмотра конфигурации
   — Статус последней операции регистрации

c) UiStore — глобальное состояние интерфейса:
   — Состояние загрузки (loading indicators)
   — Уведомления (notifications)
   — Состояние модальных окон

Преимущества Pinia в контексте PONHub:
   — Модульность: каждое хранилище отвечает за свою предметную область
   — Type safety: поддержка TypeScript для строгой типизации
   — Devtools: интеграция с Vue DevTools для отладки
   — Лёгкость тестирования: хранилища можно тестировать изолированно

3.4. Маршрутизация и навигация

Vue Router обеспечивает навигацию между страницами приложения без
перезагрузки. В PONHub реализованы следующие маршруты:

   — / — Главная панель (Dashboard): обзор всех OLT, статистика
   — /register — Регистрация новой ONU: форма ввода, предпросмотр,
     подтверждение
   — /monitor — Мониторинг ONU: таблица зарегистрированных устройств,
     MAC-адреса клиентов
   — /settings — Настройки: управление конфигурацией OLT

Навигационные переходы (Navigation Guards) обеспечивают:
   — Проверку доступности OLT перед переходом к регистрации
   — Предотвращение потери несохранённых данных
   — Перенаправление на страницу ошибок при проблемах с API

3.5. Взаимодействие с Backend API

Frontend взаимодействует с Backend через REST API, используя HTTP-запросы.
Ключевые эндпоинты:

   — POST /api/preview-onu-config — отправка данных ONU для генерации
     конфигурации, получение CLI-команд для предпросмотра
   — POST /api/register-onu — подтверждение и выполнение регистрации
   — GET /api/olt/status — получение статуса OLT-узлов
   — GET /api/onu/list — получение списка зарегистрированных ONU
   — GET /api/utils/free-ip — запрос свободного IP-адреса из БД

Обработка запросов реализована через Axios (или fetch API) с:
   — Автоматической подстановкой базового URL из переменных окружения
   — Обработкой ошибок (сетевые ошибки, ошибки сервера, таймауты)
   — Отображением индикаторов загрузки во время выполнения запросов
   — Повторными попытками при временных ошибках

4. ИННОВАЦИОННОСТЬ И НАУЧНАЯ НОВИЗНА FRONTEND-ЧАСТИ
--------------------------------------------------------------------------------

4.1. Визуализация процесса регистрации ONU

Одной из ключевых инноваций PONHub является визуализация многоступенчатого
процесса регистрации ONU. В традиционных системах оператор вводит команды
и получает результат без промежуточного контроля. PONHub реализует
поэтапный процесс:

Этап 1: Ввод данных — пользователь вводит SN, тип ONU, VLAN
Этап 2: Обогащение — система автоматически запрашивает свободный IP из БД
         и свободный ONU-ID с OLT
Этап 3: Генерация — на основе Jinja2-шаблонов создаются CLI-команды
Этап 4: Предпросмотр — пользователь ВИДИТ все команды до их выполнения
Этап 5: Подтверждение — пользователь подтверждает выполнение
Этап 6: Результат — система отображает результат выполнения каждой команды

Такой подход, основанный на принципе "четырёх глаз" (four-eyes principle),
широко используемом в банковском секторе и критических системах, впервые
применён в контексте управления абонентским оборудованием GPON.

4.2. Шаблонно-ориентированная архитектура конфигурации

Frontend PONHub не содержит жёстко закодированных CLI-команд. Вместо этого
он передаёт структурированные данные (SN, тип, VLAN, IP, ONU-ID) на
Backend, который использует Jinja2-шаблоны для генерации команд. Это
обеспечивает:

   — Вендоронезависимость: шаблоны можно создавать для разных производителей
     (ZTE, Huawei, Nokia) без изменения Frontend
   — Версионирование: шаблоны можно версионировать и отслеживать изменения
   — Аудит: по конфигурации можно определить, какой шаблон использовался

Frontend обеспечивает корректную визуализацию результатов шаблонизации,
группируя команды по функциональным блокам и выделяя критические операции.

4.3. Реактивный мониторинг в реальном времени

Frontend PONHub реализует периодический опрос Backend для получения
актуальной информации о состоянии ONU и MAC-адресах клиентских устройств.
Это позволяет оператору:

   — Контролировать работоспособность подключённых ONU
   — Диагностировать проблемы (отсутствие связи, потеря пакетов)
   — Видеть MAC-адреса устройств за ONU (для NAT/бридж-режимов)

Реализация через Pinia + реактивные компоненты Vue.js обеспечивает
автоматическое обновление интерфейса без действий пользователя.

5. ПРАКТИЧЕСКАЯ ЦЕННОСТЬ И РЕЗУЛЬТАТЫ ВНЕДРЕНИЯ
--------------------------------------------------------------------------------

5.1. Количественные показатели эффективности

Внедрение PONHub обеспечивает следующие улучшения по сравнению с ручным
процессом:

Показатель                    | Ручной процесс | PONHub      | Улучшение
------------------------------|----------------|-------------|----------
Время регистрации 1 ONU       | 10–20 мин      | 2–3 мин     | 80–85%
Вероятность ошибки конфигурации| ~15%          | <2%         | 87%
Количество CLI-команд вручную | 15–25          | 0           | 100%
Необходимость экспертных знаний| Высокая        | Базовая     | Значительное

5.2. Качественные преимущества

1) Снижение требований к квалификации персонала. Оператору не нужно знать
   точный синтаксис CLI-команд ZTE OLT. Достаточно ввести серийный номер,
   тип устройства и VLAN.

2) Документирование и аудируемость. Каждая операция регистрации фиксируется
   в системе, что позволяет проводить аудит и расследование инцидентов.

3) Масштабируемость. Единый интерфейс для управления несколькими OLT
   позволяет оператору обслуживать больше абонентов без роста штата.

4) Обучение новых сотрудников. Интуитивный интерфейс PONHub сокращает
   время обучения нового оператора с нескольких недель до нескольких дней.

5.3. Адаптивность и кроссплатформенность

Благодаря веб-ориентированной архитектуре PONHub доступен с любого
устройства, имеющего браузер: настольный компьютер, планшет, смартфон.
Это обеспечивает гибкость работы инженеров, в том числе удалённо.

Bootstrap 5 обеспечивает адаптивную вёрстку, автоматически подстраивая
расположение элементов под размер экрана.

6. ПЕРСПЕКТИВЫ РАЗВИТИЯ FRONTEND-ЧАСТИ
--------------------------------------------------------------------------------

6.1. Поддержка темной темы и кастомизация интерфейса

Планируется внедрение системы тем (светлая/тёмная) с использованием CSS
Custom Properties (CSS Variables), что позволит пользователям выбирать
комфортный режим работы, особенно важный для круглосуточных операционных
центров (NOC — Network Operations Center).

6.2. Визуализация топологии сети

Разработка интерактивной карты топологии сети GPON с отображением:
   — OLT-узлов и их взаимосвязей
   — PON-портов и подключённых ONU
   — Статусов соединений (цветовая индикация)
   — Загрузки каналов

Реализация планируется с использованием библиотеки D3.js или ECharts для
отрисовки графов.

6.3. Интеграция с системами тикетов и мониторинга

Frontend может быть расширен для интеграции с внешними системами:
   — Создание тикетов при регистрации ONU (интеграция с Jira, Redmine)
   — Отображение алертов из систем мониторинга (Zabbix, Prometheus)
   — Отправка уведомлений в мессенджеры (Telegram, Slack)

6.4. Мобильное приложение на базе Capacitor/Ionic

На базе существующего Vue.js-кода возможно создание нативного мобильного
приложения для iOS/Android с использованием фреймворка Capacitor, что
обеспечит работу инженеров "в поле" без необходимости доступа к ноутбуку.

6.5. PWA (Progressive Web App)

Добавление поддержки PWA позволит:
   — Устанавливать приложение на рабочий стол
   — Работать в оффлайн-режиме (кэширование данных)
   — Получать push-уведомления о статусе операций

--------------------------------------------------------------------------------
7. ЗАКЛЮЧЕНИЕ
--------------------------------------------------------------------------------

Frontend-часть системы PONHub представляет собой современное веб-приложение,
построенное на базе Vue.js 3, Pinia, Vue Router и Bootstrap 5. Архитектура
приложения обеспечивает:

1) Интуитивный пользовательский интерфейс для регистрации и управления ONU
   в сетях GPON без необходимости знания CLI-команд.

2) Поэтапный процесс с предварительным просмотром конфигурации, реализующий
   принцип "четырёх глаз" для минимизации ошибок.

3) Реактивный мониторинг состояния оборудования в реальном времени.

4) Модульную компонентную архитектуру, обеспечивающую лёгкое расширение и
   поддержку нового функционала.

5) Кроссплатформенную доступность через браузер.

Научная новизна заключается в применении принципов веб-ориентированной
автоматизации к задаче регистрации абонентского оборудования GPON,
визуализации процесса генерации конфигурации через шаблонизатор и
реактивном мониторинге сетевых устройств.

Практическая ценность подтверждается сокращением времени регистрации ONU
на 80–85%, снижением вероятности ошибок на 87% и уменьшением требований
к квалификации обслуживающего персонала.

Проект PONHub демонстрирует, как современные веб-технологии могут быть
применены для решения задач телекоммуникационной отрасли, обеспечивая
высокую эффективность, надёжность и удобство использования.
